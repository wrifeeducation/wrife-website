What WriFe Is

WriFe is a UK school writing platform for pupils aged 6–14 (Years 2–8, KS1–KS3). It combines structured grammar-based writing instruction with gamified practice, AI-assisted assessment, and a professional teacher resource toolkit. The platform is built across five distinct applications that share a single Supabase backend.

The Golden Rule All five apps share exactly one Supabase project: gzmgjkbtsvezfclmreru (WriFe Platform). Never create migrations targeting any other project ID for production work. The two legacy project IDs (rxmitjrbrsqjeymsycoj, nxhkpqngnxshgotvuujb) are retired and should be deleted to avoid confusion (see Data Architecture tab).

The Four Login Routes

Understanding which route a user arrives through determines their interface, session flags, and data access. All routes authenticate against the single Supabase project.

Route A — School Pupil Hub SSO ✅

All school pupils. The ONLY valid login path for school-managed accounts.

Pupil visits wrife.co.uk/pupil/login
Enters class_code + username + PIN
POST /api/pupil/login → validates against pupils table
Supabase Auth user provisioned (synthetic email: pupil-{uuid}@practice.wrife.co.uk)
JWT minted → pupil dashboard at /pupil/dashboard
App tile clicked → JWT embedded in URL hash fragment
Sub-app SDK auto-detects hash → setSession() called
sessionStorage.setItem('entryViaHub', '1')
"← WriFe" back button shown in sub-app

Route B — Teacher / Admin Email Login ✅

School teachers, school admins, super admin, independent teacher pupils, home learners.

Teacher → wrife.co.uk/login (email + password)
Admin → wrife.co.uk/admin/login (special portal)
Supabase Auth cookie session created
Profile fetched → role read from profiles.role
Routed to appropriate dashboard
⚠️ Route B for school pupils on sub-apps is RETIRED — sub-app /login must reject school pupils and redirect to wrife.co.uk

Route C — Parent Home Sign-up ✅ (partial)

B2C parents purchasing access for their children.

Parent visits pwp-studio.wrife.co.uk/home-signup
Account created in home_accounts (account_type='parent')
Stripe checkout → subscription attached
Post-payment: create-child-profile Edge Function provisions pupil
Auto-created home class with parent code (= class code)
Child logs in at sub-app /login using parent code + username + PIN
⚠️ Stripe webhook → pupil provisioning not yet fully built

Route D — Independent Teacher Sign-up ❌ Not Built

Teachers with no school account who sign up directly on sub-apps.

Teacher signs up at pwp-studio.wrife.co.uk/teacher-signup or practice.wrife.co.uk/teacher-signup
Account in home_accounts (account_type='independent_teacher')
Stripe subscription attached
Creates classes in sub-app → pupils login via Route B with class code
All of the above is yet to be built on any sub-app

The ← WriFe Back Button

Shown in sub-apps only when the pupil arrived via Route A (hub hash token). Never shown for direct logins.

// Run once on app init (both PWP Studio and Interactive Practice):
supabase.auth.onAuthStateChange((event, session) => {
  if (event === 'SIGNED_IN' && window.location.hash.includes('access_token')) {
    sessionStorage.setItem('entryViaHub', '1')
    window.history.replaceState(null, '', window.location.pathname)
  }
})

// In nav component:
const showBackToHub = sessionStorage.getItem('entryViaHub') === '1'
// Render: <a href="https://wrife.co.uk/pupil/dashboard">← WriFe</a>

Use sessionStorage (not localStorage) — clears on tab close so a fresh direct load never incorrectly shows the button.

The learning_events Bridge

Sub-apps write a lightweight event row whenever significant pupil progress occurs. wrife.co.uk reads this table for the teacher class view (ClassActivityPanel). Sub-app internal tables remain private.

CREATE TABLE learning_events (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  pupil_id     UUID NOT NULL,       -- = auth.uid() = pupils.id
  app          TEXT NOT NULL CHECK (app IN ('pwp', 'ip', 'dwp', 'resources')),
  event_type   TEXT NOT NULL,
  event_data   JSONB DEFAULT '{}',
  class_id     UUID REFERENCES classes(id),  -- nullable for home learners
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

  • home_accounts uses home_account_id (not owner_id) for class ownership

The WriFe Writing Curriculum

68 lesson items (L1–L68, where L27 splits into L27a and L27b) organised into 7 chapters and 16 units. Designed for UK pupils across Key Stages 1–3 (Years 2–8, ages 6–14). The curriculum spans primary and lower secondary — WriFe is not limited to primary only. PWP formula practice begins at Lesson 10. The final lesson, L68 "Celebrating the End Results", is a project celebration activity.

Core Governance Rules

Rule 1 — Stay in your lane When working in wrifeapp: only create/alter PWP-owned tables. When working in InteractivePracticeApp: only create/alter IP-owned tables. When working in wrife-dwp: only create/alter DWP-owned tables. When working in wrife-website: you may alter any table, but flag changes to shared tables explicitly.
Rule 2 — Cross-repo changes require an explicit plan Complete in this order: (1) wrife-website schema/API changes first, (2) sub-app consumes the new schema, (3) verify cross-app checklist before merging either.
Rule 3 — When wrife.co.uk needs updating from a sub-app change A sub-app change triggers a wrife.co.uk change ONLY when: a new assignment type is needed on the teacher dashboard; a new data point needs to appear in the teacher class view; the pupil dashboard SSO tile needs updating; a new learning_events event_type is introduced (document it in the Data Architecture tab first). All other sub-app changes are self-contained.
Rule 4 — Standalone mode always works Both sub-apps must work when class_id is null (home learners). Every new progress/session table must have class_id nullable. Every upsert must use pupil_id as the sole conflict key when class_id is null.
Rule 5 — Feature parity Any access pattern supported by one sub-app must be supported by both. Do not build a flow (e.g. home sign-up) in one sub-app and leave it missing from others.

Architecture Decision Log

Significant decisions that should not be re-opened without a deliberate review. Each entry records what was decided, why, and the implications for future work.

ADR-001 · 2026-03 · Auth

Route B retired for school pupils on sub-apps

Decision: School pupils must ONLY log in via wrife.co.uk (Route A). Sub-app /login pages must detect school pupils and redirect them to wrife.co.uk.

Why: Having two login paths for school pupils caused session confusion and stale localStorage issues. Pupils who logged in directly bypassed the hub SSO and ended up with incomplete session state that broke the "← WriFe" back-button and class context.

Implication: Any new sub-app login form must include a guard: if the pupil's class is school-type, reject and redirect.

ADR-002 · 2026-03 · Database

Single Supabase project for all four apps

Decision: All apps share gzmgjkbtsvezfclmreru. The two previously isolated projects (rxmitjrbrsqjeymsycoj, nxhkpqngnxshgotvuujb) are retired.

Why: Three isolated databases made cross-app features (teacher visibility, SSO, learning_events) impossible without expensive API bridges. A single project enables RLS-secured cross-app queries and a unified auth token.

Implication: Every migration must target gzmgjkbtsvezfclmreru only. Table ownership rules (see Data Architecture tab) exist to prevent chaos.

ADR-003 · 2026-04 · Data

formula_progress is the primary PWP dashboard table

Decision: All new gamification columns (XP, coins, streaks, badges) must go on formula_progress, not on pwp_pupil_progress.

Why: DashboardPage.tsx in wrifeapp queries formula_progress. pwp_pupil_progress is a legacy/parallel table that is not currently queried by the dashboard UI. Adding columns there caused them to silently not appear.

Implication: Any new column must use NOT NULL DEFAULT <value> to backfill existing rows. A column without a default will cause NULL crashes for all existing pupils on next dashboard load.

ADR-004 · 2026-04 · Database

DWP RLS uses auth_user_id join pattern

Decision: DWP RLS policies must use pupil_id IN (SELECT id FROM pupils WHERE auth_user_id = auth.uid()).

Why: The pupils.id is not the same as auth.uid() for DWP. Using pupil_id = auth.uid() directly returns no rows because DWP creates pupil records with their own UUID that doesn't match the auth user ID.

Implication: Every new DWP table that RLS-protects pupil data must use this join pattern, not the simple equality check.

ADR-005 · 2026-05-13 · Repo boundary

wrife-website repo boundary enforced after mistaken PWP build

Decision: The wrife-website repo is for wrife.co.uk only. No PWP pupil experience, no IP game code, no DWP writing app. Any such code must be surgically removed.

Why: A full PWP implementation (app/pwp/ routes + components/pwp2/) was built inside wrife-website by mistake over a full session. It required a git reset --hard 375d29d to remove. Wasted a full day.

Implication: At the start of any session touching wrife-website, confirm the task is about wrife.co.uk features (teacher dashboard, admin, marketing). If the task is about PWP/IP/DWP pupil experience, stop and connect the correct repo.

ADR-006 · 2026-05 · Infrastructure

DB connection pool max: 2 on free-tier Supabase

Decision: lib/db.ts in wrife-website must keep max: 2 (with idleTimeoutMillis: 10000, connectionTimeoutMillis: 8000).

Why: Vercel serverless functions each spawn a new connection pool. The original max: 10 setting exhausted Supabase's free-tier ~100 connection limit at ~10 concurrent users, triggering a cascade of 504 errors on the teacher dashboard.

Implication: When upgrading to Supabase Pro, switch the connection string to the Session Mode pooler URL (port 5432) to remove the ceiling. Until then, max: 2 is non-negotiable.

ADR-007 · 2026-05 · Infrastructure

Workbox NetworkOnly rules required for Supabase endpoints in wrifeapp

Decision: wrifeapp/vite.config.ts must have NetworkOnly rules for /functions/ and /auth/ before any NetworkFirst REST rule.

Why: The PWP Studio service worker was intercepting and serving cached responses for Supabase Edge Function and auth calls. This caused silent failures where pupils saw stale data or got auth errors even with valid sessions.

Implication: Any new Supabase endpoint pattern added to wrifeapp must also be added to the NetworkOnly list before deployment.

ADR-008 · 2026-05 · Cross-app

learning_events CORS requirement: apikey header mandatory

Decision: All WriFe Edge Functions must include apikey in their CORS Access-Control-Allow-Headers list.

Why: Missing apikey caused a silent TypeError: Failed to fetch with no useful error message in the console. The DWP Edge Functions were missing this header and silently failing for all cross-origin calls from the DWP SPA.

Implication: The full required header list is: authorization, x-client-info, apikey, content-type. Validate this in the auth sweep skill after every Edge Function deployment.

ADR-009 · 2026-05 · DWP

DWP shares the platform Supabase project (not isolated)

Decision: DWP tables were created in gzmgjkbtsvezfclmreru, not in a separate DWP project.

Why: Following ADR-002. DWP needs access to pupils, classes, and learning_events which are all owned by wrife-website. Isolation would require API bridges.

Implication: DWP's home_accounts_app_origin_check constraint must include 'dwp'. The DWP RLS pattern (ADR-004) applies to all DWP pupil tables.

ADR-010 · 2026-05 · Platform

resources.wrife.co.uk joins as the 5th app on the shared Supabase project

Decision: The Learning Toolkit (resources.wrife.co.uk) uses gzmgjkbtsvezfclmreru, matching all other apps. It owns its own tables (ai_sessions, ai_attempts, daily_streaks, usage_quotas, ai_tool_assignments) and writes to learning_events with app = 'resources'.

Why: Consistent with ADR-002. Resources needs access to pupils and classes for school-tier licence verification. Sharing the project enables the same RLS-secured patterns used by all other apps.

Implication: The learning_events.app CHECK constraint must be updated to include 'resources' before the app goes live. The home_accounts_app_origin_check constraint may also need updating if resources.wrife ever creates home accounts. Two pending migrations required (see Platform Status tab, items 7–8).

ADR-011 · 2026-05 · Curriculum

WriFe curriculum covers KS1–KS3 (Years 2–8), not primary only

Decision: The authoritative curriculum scope is Key Stages 1–3, Years 2–8 (ages 6–14). Lessons L27–L68 are designated for Years 5–8 (KS2 upper / KS3). Platform documentation and database year group fields should reflect this range.

Why: Previous documentation incorrectly capped the curriculum at Years 4–5 (ages 8–10). The WriFe Bible ("Writing for Everyone") and the platform designer's explicit guidance confirm the curriculum extends into secondary (Year 7–8 for lessons L27+). WriFe is designed for the full transition from primary into lower secondary.

Implication: The year_group_max values in the lessons table for L27+ need correcting. Any marketing copy, teacher documentation, or public-facing material referencing "primary only" should be updated. The platform subtitle and PAD Overview tab have been updated to reflect KS1–KS3.
