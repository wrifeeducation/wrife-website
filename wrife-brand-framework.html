Chunky CTA Buttons

The WriFe World button has a thick bottom border that creates physical 3D depth. On tap/click it physically presses down (translateY + box-shadow collapse). Never use CSS gradients or glow effects on buttons.

/* The bottom-border "press" pattern (Framer Motion) */ style={{ background: '#F5A623', boxShadow: '0 4px 0 0 #c47a0a', // thick bottom border }} whileHover={{ scale: 1.01 }} whileTap={{ y: 4, boxShadow: '0 0 0 0 transparent' }} // presses down

Touch target minimum: 52px height (desktop) / 56px height (mobile). Use min-height: 52px always.

Chapter Banners

Bold full-bleed banners introduce each chapter or unit. The thick bottom border (4px solid darker shade) reinforces the physical language of the app. White text only. Pills use semi-transparent white backgrounds.

Feedback Panels

Feedback slides up from the bottom of the screen as a full-width fixed panel (Framer Motion: initial={{ y: '100%' }} → animate={{ y: 0 }}, spring). Two states: correct (teal) and needs revision (orange). The Continue button is always white.

Do / Don't

Key guardrails for consistent WriFe UI.

✅ DO — Use chunky bottom-border buttons

Apply boxShadow: '0 4px 0 0 {darkerShade}' and whileTap={{ y: 4, boxShadow: '0 0 0 0 transparent' }} on every primary CTA.

❌ DON'T — Use CSS gradients or glow effects

No linear-gradient, no box-shadow with blur or spread for glow. WriFe buttons are flat + thick bottom border only.

✅ DO — Use fluid type tokens

Always reference font-size: var(--pwp-text-base) etc. The clamp function handles responsive scaling automatically.

❌ DON'T — Hardcode font sizes in px

Never write font-size: 16px directly. This breaks the fluid scale and creates inconsistency across screen sizes.

✅ DO — Use canonical word-class colours

Import from wordClassColours.ts. Noun = coral red, Verb = royal blue, etc. These never change.

❌ DON'T — Redefine word-class colours elsewhere

The colours are a single source of truth. Defining them in component CSS creates drift and breaks the coherent colour grammar that pupils learn to associate with each class.

✅ DO — Slide feedback up from the bottom

Use position: fixed; bottom: 0 with a spring animation. This mirrors Duolingo's proven UX pattern and keeps the learning flow uninterrupted.

❌ DON'T — Inline feedback inside the step card

Inline feedback was the old pattern. It caused layout shifts and broke the visual hierarchy. Always use the full-width bottom panel.
